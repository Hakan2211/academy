<Step title="From lines to hierarchies" kind="explain">

Arrays, stacks, queues and linked lists are all **linear** — one thing after another. But much of the world is **hierarchical**: a family tree, the folders on your computer, the chapters-and-sections of a book, the management chart of a company. To capture "this belongs under that," we need a **tree**.

A tree is made of **nodes** connected by edges, with a special vocabulary:

- the top node is the **root**,
- a node directly above another is its **parent**; below is a **child**,
- a node with no children is a **leaf**.

Crucially, a tree never loops back on itself — there's exactly one path from the root to any node.

</Step>

<Step title="Binary search trees" kind="explain">

A **binary** tree gives each node at most **two** children — a left and a right. The real magic comes from one extra rule that makes a **binary search tree (BST)**:

> For every node: everything in its **left** subtree is **smaller**, everything in its **right** subtree is **larger**.

That single ordering rule means the tree is always sorted, and it turns searching into a fast game of "higher or lower."

</Step>

<Step title="Grow one, then search it" kind="interactive">

Insert values below. Each new value starts at the root and walks down: go **left** if it's smaller, **right** if it's larger, until it finds an empty spot. Then press **Search** and watch the path light up from the root to the target.

<TreeViz />

Notice the search never wanders. At each node it compares once and **discards an entire subtree** — half the remaining values gone in one step. In a balanced tree that means a lookup takes only about <Formula tex="O(\log n)" block /> steps: double the data, and you add just **one** more level to search.

</Step>

<Step title="Why log-depth matters" kind="explain">

That halving is the same superpower you saw in **binary search** last world — but now it's baked into the *structure* itself, not just one clever algorithm. A balanced BST with a million values is only about **20** levels deep, so any lookup, insert or delete touches roughly 20 nodes.

<Callout type="warn" title="Balance is everything">
The <Formula tex="O(\log n)" block /> promise assumes the tree stays **balanced** — bushy, not lopsided. Insert already-sorted values into a plain BST and it degenerates into a glorified linked list, one long chain, dragging search back to <Formula tex="O(n)" block />. Real systems use **self-balancing** trees (like red-black or AVL trees) that quietly rearrange themselves to keep the depth low.
</Callout>

Trees power database indexes, file systems, autocomplete, decision-making in games, and the parse trees compilers build from your code.

</Step>

<Step title="Recap" kind="recap">

- A **tree** is a hierarchy of **nodes**: a **root** at top, **parent/child** links, and **leaves** at the ends — no loops.
- A **binary search tree** gives each node up to two children with the rule **smaller left, larger right**.
- That ordering makes search a "higher or lower" game that discards half the tree each step: <Formula tex="O(\log n)" block /> when balanced.
- Stay **balanced** or the tree degrades to a chain and search slips to <Formula tex="O(n)" block />.
- Trees underlie file systems, database indexes, autocomplete and compilers.

</Step>
