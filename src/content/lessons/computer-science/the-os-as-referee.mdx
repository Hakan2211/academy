<Step title="The impartial manager in the middle" kind="explain">

We've met the operating system's services one at a time: scheduling the CPU, managing memory, handling files and devices, enforcing security. In this final lesson we tie them together around a single idea that captures what an OS *is*.

The operating system is a **referee**. On a computer, programs and users are competitors — each wanting the CPU, the memory, the disk, the network, *now*, and as much as possible. Left to themselves they would collide, cheat, or stall. The OS stands between them as the impartial authority: it grants resources, enforces the rules, settles disputes, and ensures that no competitor can harm another. Every part of this world has been one facet of that refereeing.

</Step>

<Step title="Arbitrating the CPU" kind="interactive">

Start with the most contested resource of all: time on the CPU. Many processes are Ready; only one can run per core. The scheduler is the referee blowing the whistle — deciding whose turn it is, for how long, and when to cut someone off.

<Scheduler />

Switch the algorithm and watch the verdict change. This is refereeing in its purest form: the same competitors, the same scarce resource, but different *policies* produce different outcomes for fairness, responsiveness and average wait. There's no single "right" call — only a policy the OS commits to and enforces consistently, the same way a referee applies one rulebook evenly to both teams.

</Step>

<Step title="Arbitrating everything else" kind="interactive">

The CPU is just one arena. The same refereeing extends to every shared resource on the machine — review the OS's services and notice that each is really a *rule about sharing something fairly and safely*.

<OsServices />

Memory management referees RAM, handing each process a private space and blocking trespass. The file system referees the disk, so two programs writing at once don't scramble each other's data. Device drivers referee the keyboard, screen and network, queueing requests so they don't conflict. Security referees *users*, deciding who may touch what. In every case the pattern is identical: a scarce, shared resource, many competitors, and the OS as the trusted authority allocating it.

</Step>

<Step title="The power to enforce: user mode and kernel mode" kind="explain">

A referee is useless without authority. What gives the OS the power to *enforce* its rulings, even against a buggy or malicious program?

The answer is built into the CPU hardware itself: it runs in one of (at least) two **privilege levels**. Ordinary programs run in **user mode**, where the most dangerous instructions — directly touching hardware, editing the page tables, halting the machine — are simply **forbidden**. The OS kernel runs in **kernel mode**, where everything is permitted. A user-mode program that *tries* a privileged instruction is instantly stopped by the hardware and handed to the kernel to deal with.

<Callout type="info" title="Why two modes change everything">
This single hardware distinction is what makes a safe, shared computer possible. Because apps physically *cannot* bypass the OS, a crashing or hostile program can wreck only itself — not the kernel, not other programs, not the machine. The referee can't be shoved off the field.
</Callout>

</Step>

<Step title="Crossing the line: system calls" kind="explain">

If apps live in user mode and can't touch hardware, how do they ever get anything done — open a file, send data, draw a window? Through a guarded gateway called a **system call**.

When a program needs a privileged service, it doesn't *do* the action; it *asks*. A system call triggers a controlled switch into kernel mode at a fixed, OS-chosen entry point. The kernel checks the request — are you allowed to read this file? is this address yours? — performs it if legitimate, and switches back to user mode with the result. The boundary is crossed only at the gate, only with the kernel's permission, and only on the kernel's terms. This is exactly the **abstraction** idea from the start of the island, now enforced by hardware: a simple, safe interface over a dangerous, powerful reality.

</Step>

<Step title="Protection turns sharing into safety" kind="explain">

Put the pieces together and you see why a modern computer can safely run code you'd never trust on its own — a random website's JavaScript, a downloaded game, dozens of apps from dozens of authors — all at once, on the same hardware as your bank login.

Each runs as a process in **user mode**, fenced into its **own virtual address space**, given turns by the **scheduler**, and able to reach the hardware only through checked **system calls**. The kernel, in **kernel mode**, is the referee with both the *policy* (who gets what) and the *power* (hardware-enforced privilege) to make it stick. Multitasking isn't just convenient — protection is what makes it *safe*.

</Step>

<Step title="The big picture" kind="explain">

Zoom all the way out. Back in World 4, a computer was one CPU honestly fetching and executing machine code, one instruction at a time. That literal machine hasn't changed. What the operating system adds is a layer of **illusion and order** on top of it: the illusion that there are many CPUs (time-slicing), that memory is vast and private (virtual memory), that the disk is full of tidy files, that many users and programs coexist without conflict.

Every one of those illusions is really the same two jobs you met in lesson one — **abstraction** and **resource management** — now scaled up to govern the whole system. The OS is the master program that turns a single, literal machine into a safe, fair, shared, multitasking computer. It is the referee that makes everything above it possible.

</Step>

<Step title="Recap" kind="recap">

- The OS is an impartial **referee**, arbitrating scarce, shared resources among competing programs and users.
- It referees the **CPU** (scheduling), **memory** (virtual address spaces), the **disk** (file system), **devices** (drivers) and **users** (security) — all the same pattern of fair, safe sharing.
- The CPU's **user mode / kernel mode** split gives the OS hardware-enforced authority: apps can't run privileged instructions or touch hardware directly.
- Apps reach privileged services only through guarded **system calls**, which the kernel checks before performing — abstraction enforced by hardware.
- **Protection** is what makes multitasking *safe*: each process is fenced into its own space and given turns, so untrusted code can run beside your most sensitive data.

</Step>
